## 设计哲学 · Two parallel lines

Daemonkey 的载体演进是**两条并行线**，缺一不可：

```
                         本地 AI 搭档 (daemon)
                                  │
                  ┌───────────────┴───────────────┐
                  │                               │
        ┌─────────▼─────────┐           ┌─────────▼─────────┐
        │  物理在场线         │           │  远程指挥线         │
        │  (它的肌肉)         │           │  (你的耳麦)         │
        ├───────────────────┤           ├───────────────────┤
        │ 桌宠 / 副屏        │           │ 网页 WebUI ✅      │
        │ 桌面机器人          │           │ 微信 / IM          │
        │ (树莓派+触屏+语音)  │           │ Telegram bridge    │
        └───────────────────┘           └───────────────────┘
```

- **只有物理在场没有远程** → 你出门就跟它断联
- **只有远程没有物理在场** → 你不在时它没法真正"在场"陪你

**不变的内核**：可换模型、可换设备、可换形态 —— 它始终是同一个"它"，记忆一根毫毛都不丢。

## 🌱 近期 · Near-term

> 把当前形态打磨扎实。

- **记忆体系深化** —— 跨会话回忆（`session_search`：把历史对话也纳入 FTS5 索引），让"突然想起某次聊过的事"成为可能。
- **技能系统完善** —— 主动沉淀（跑过 N 次后提示「要不要记成 playbook」）+ Progressive Disclosure（按需注入，不撑爆上下文）。
- **完整 WebUI 工具链** —— 把更多能力从开发主线增量搬入用户版。
- **稳定性 & 体验** —— 持续打磨启动、升级、自愈链路。

---

## 🚀 中期 · Mid-term

> 多载体 + 能力生长。

- **终端形态** —— 命令行 REPL，给习惯终端的用户。
- **远程指挥线** —— 微信 / IM 入口（等通用友好协议）、Telegram bridge 补位，让你在外也能指挥它。
- **第三方 API 应用** —— 把任意服务商（图像 / 语音 / 视频生成等）的 KEY + 文档喂给它，它封装成一张工坊卡片。
- **应用串成生产线** —— 工坊 App 通过工作流串成自动化流水线。
- **自演化深化** —— 它不只随你成长，也能反思自己、主动提议改进自己的设定（你确认后才合入）。

---

## 🌌 远期 · Long-term

> 物理在场 + 永续陪伴。

- **桌面机器人** —— 树莓派 / 触摸屏 / 麦克风 / LED：屏幕上随时显示它在做什么，语音唤醒，情绪表达。
- **多设备同步** —— 手机 + 电脑 + 桌面机器人，会话与记忆实时同步（L6 跨容器同身）。
- **本地 LLM 兜底** —— 断网时自动 fallback 到本地模型。
- **长期归档** —— 灵魂套件自动版本化、可继承。

---

## ❌ 不做的事 · Non-goals（防 scope creep）

- **不做闭源商用 SaaS** —— Daemonkey 是属于你的搭档，不是托管服务。AGPL-3.0 守住这条。
- **不绑单一 provider** —— 所有模型知识集中在一处，新接一家只加白名单几行；目的是摆脱单点失败。
- **不做技能应用商店** —— 能力挂在你这一个搭档身上，社区共建走 **PR 回内核**，让每个改进惠及所有人的搭档。
- **不做花哨重前端框架** —— 够用、快、稳优先。
